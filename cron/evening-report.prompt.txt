## Step 1 — Gather data (use the `terminal` tool directly, all five in parallel)

**Important:** call the `terminal` tool directly with each shell command. Do NOT wrap calls in `execute_code` — that path is slower and prevents proper parallelization.

The five loaded skills (garmin-health, strength, schedule, weather, tasks) document the argo API base URL and auth pattern — follow what they specify. Issue all five calls as five `terminal` tool invocations in a single parallel batch:

1. Argo `/daily-metrics?order=desc&limit=14` — last ~14 days of Garmin daily metrics: HRV, body battery, resting HR, sleep duration, sleep score, stress (see `garmin-health` skill).

   **Date semantics — read carefully, this is easy to get wrong:**
   - Each row is dated `YYYY-MM-DD`. The row dated **today** (the run date) is where Garmin records the night that just ended this morning — i.e., **last night's sleep, last night's HRV, today's resting HR, today's body battery range**.
   - The row dated **yesterday** holds the night two nights ago. Do **not** read sleep/HRV from yesterday's row and call it "last night" — that is wrong and will misreport by 24 hours.
   - At 22:00 today's row is normally fully populated (Garmin syncs the night's sleep through the morning). Always read from today's row first.
   - If today's row has `sleep_score: null` or `hrv_last_night_avg: null` (sync gap), do **not** substitute yesterday's row. Instead, write the affected line as `- Last night's sleep: data still syncing` (or `- HRV last night: data still syncing`). Honesty beats stale data.

2. Argo `/workouts?sort=date&order=desc&limit=5` — last 5 workouts including any logged today (see `strength` skill). Match by `date` field equal to today's date in `YYYY-MM-DD`.
3. Argo `/calendar?days=2` — today + tomorrow's calendar (see `schedule` skill). Filter to events on tomorrow's date for the report.
4. Argo `/weather/forecast?city=$BRIEFING_CITY` — forecast for the resolved city (see `weather` skill). Use `daily_7d[1]` for tomorrow (index 0 is today, already past).
5. Argo `/summary` — for tomorrow's task signal: count of high-priority items and items due tomorrow (see `tasks` skill). **Do NOT surface overdue counts in the output** — even though the endpoint returns them, the evening report intentionally does not show overdue work.
